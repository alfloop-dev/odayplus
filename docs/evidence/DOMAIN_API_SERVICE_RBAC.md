# Domain API Service & Server-Side RBAC (ODP-GAP-API-001)

Status: delivered · Owner: Claude2 · Reviewer: Claude
Source: `docs/evidence/PROJECT_WIDE_GAP_AUDIT.md`, ODP-SD-06 (API design),
ODP-SD-08 (workflow/state), ODP-SD-09 (identity/security).

## What this delivers

The FastAPI domain API (`apps/api/oday_api/main.py::create_app`) exposes the
integration / opsboard / data / ML libraries as HTTP endpoints, wired to durable
repositories and stores — not seed JSON — and guarded by server-side RBAC.

| Concern | Where |
| --- | --- |
| App entrypoint, health/version, jobs, audit feed | `apps/api/oday_api/main.py` |
| Real repositories / stores (in-memory or durable SQLite) | `shared/infrastructure/persistence.build_persistence()` (ODP-PV-009) |
| Domain routers | `apps/api/oday_api/routes/heatzone.py`, `apps/api/app/routes/{avm,forecastops,priceops,netplan,interventions,sitescore,adlift,audit,learninghub,listings,external_data}.py` |
| RBAC route guard | `apps/api/oday_api/security/dependencies.require_permission` |
| Role → permission matrix | `shared/auth/rbac.py` (ODP-SA-04 §6) |
| OpenAPI contract | generated by FastAPI, asserted in tests |

The endpoints are backed by the real module services (`AVMService`,
`ForecastOpsService`, `InterventionWorkflow`, `SiteScoreDecisionWorkflow`, …)
resolved from the persistence bundle, so a request executes production domain
logic and persists through the configured store.

## RBAC enforcement model (two layers)

Authorization is layered exactly as ODP-SD-09 §5 specifies, and the two layers
run at different points:

1. **Route/type-level RBAC** — `require_permission(resource, action)` is a
   FastAPI dependency on each domain route. It answers only "does the caller's
   role grant `action` on `resource`" (`rbac_allows`). A denied or anonymous
   caller gets **HTTP 403** and the denial is written to the platform security
   audit log (`security.authorization`, ODP-AC-AUTH-005).
2. **Object-level ABAC + high-risk governance** — scope containment, data
   classification, and the high-risk feature-flag / separation-of-duties hooks
   need the *target object's* attributes (tenant/region/store, proposer,
   risk level). They are evaluated inside the handler and the domain workflow
   once the object is loaded, via the full `AuthorizationEngine`.

This separation is deliberate. High-risk verbs (`approve`, `execute`,
`publish`, `override`, `rollback`) fail **closed** in the engine unless an
active high-risk feature flag and object context are present. Running the whole
engine at the *route/type* level — where no object is loaded and no flag is
enabled — would return 403 for every high-risk endpoint regardless of
credentials. Route-level guarding therefore enforces RBAC only; the high-risk
policy remains where it can see the object.

## Route → resource/action coverage

Every domain router included by `create_app` is guarded. Each router builds its
`require_permission` dependency from an engine sharing the app-wide `audit_log`,
so a 403 denial on any route writes the same `security.authorization` event
stream the `/audit/events` feed exposes.

| Router (prefix) | Read routes | Write routes | RBAC `resource` | Least-privilege role |
| --- | --- | --- | --- | --- |
| `/heatzones` | `GET ""`, `/map`, `/snapshots/{id}`, `/{h3_index}` → `view` | `POST /score-jobs` → `create` | `heatzone` | `EXPANSION_USER` |
| `/listings` | `GET /candidates` → `view` | `POST /import`, `/import-jobs` → `create` | `listing` | `EXPANSION_USER` |
| `/external-data` | `GET /freshness` → `view` | — | `integration` | `DATA_OWNER` |
| `/sitescore` | `view` | `execute` / `approve` | `sitescore` | `EXPANSION_USER` / `SITE_REVIEWER` |
| `/forecastops` | `view` | `create` / `execute` | `forecastops` | `OPERATIONS_MANAGER` |
| `/interventions` | `view` | `create` / `approve` / `execute` | `intervention` | `OPERATIONS_MANAGER` (+ `REGIONAL_SUPERVISOR`) |
| `/priceops` | `view` | `create` / `approve` / `execute` | `priceops` | `PRICING_MANAGER` |
| `/adlift` | `view` | `create` / `approve` | `adlift` | `MARKETING_MANAGER` |
| `/avm` | `view` | `create` / `approve` / `export` | `avm` | `FINANCE_LEGAL` |
| `/netplan` | `view` | `create` / `approve` / `execute` | `netplan` | `EXECUTIVE` |
| `/learninghub` | `view` | `publish` / `rollback` | `model` | `MODEL_OWNER` / `RELEASE_OWNER` |
| `/audit` | `view` | `export` | `audit` | `AUDITOR` |

The `heatzone`, `listings`, and `external_data` routers were the three that
previously shipped with **no** route guard (ODP-GAP-API-001 review). `listings`
and `external_data` were module-level routers; they are now factories
(`create_listings_router` / `create_external_data_router`) so `create_app`
threads the shared `audit_log` into their authorization engine, exactly like the
other domain routers.

## Boundary with ODP-OC-PROD-001 (console BFF)

ODP-GAP-API-001 owns the **domain data/ML/integration read-write endpoints**
listed above. ODP-OC-PROD-001 owns the **operator-console BFF** — an
aggregation/presentation layer for the console UI. There is no overlap in this
repo today: no console/BFF router exists under `apps/api/app/routes/`, and the
console BFF, when built, should compose these domain endpoints rather than
re-expose the repositories directly.
